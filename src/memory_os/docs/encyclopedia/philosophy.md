# Розділ 1: Філософія та Стратегія

Memory OS — це не просто RAG і не просто база знань. Це **локальний нейросемантичний шар довготривалої пам'яті (memory layer)** для AI-агентів. 

Система розроблена як **зовнішній гіпокамп** для великих мовних моделей (LLM). Замість того, щоб намагатися вмістити весь контекст проекту в обмежене вікно моделі, LLM використовує Memory OS як протез: вона запитує систему про релевантні фрагменти минулого досвіду, причинні зв'язки, правила та доменні знання. 

На відміну від класичних нейромереж (які вчаться через backpropagation і зберігають знання у вагах матриць), Memory OS зберігає знання явно у вигляді графа: `node → edge → metadata → evidence → lifecycle → index → retrieval`.

> **Головна концепція:** LLM не повинна памʼятати все. Вона має знати, як спитати Memory OS.

---

## 1. Філософія Local-First

Сучасні AI-агенти часто страждають від надмірної залежності від хмарних обчислень та неефективного використання LLM. Memory OS пропонує іншу формулу:

> **Формула Memory OS:**
> * **LLM говорить мало.**
> * **Memory OS робить багато.**
> * **Хмара використовується тільки тоді, коли це виправдано.**

### Ключові принципи:
1. **Локальна передобробка**: Пошук, індексування, фільтрація, класифікація, валідація, кешування та файлові операції мають виконуватися локальними скриптами, вбудованими базами даних та спеціалізованими алгоритмами на машині розробника.
2. **LLM як Диспетчер, а не Завантажувач**: Головна мета — не змушувати велику модель читати сотні кілобайт сирого коду при кожному запиті, а підготувати для неї максимально стислий, точний та релевантний контекст.
3. **Економія Токенів та Захист від Bloat**: LLM отримує лише вилучену "есенцію" проблеми. Це знижує вартість запитів, зменшує затримку (latency) та запобігає ефекту "забування" інформації всередині занадто великих контекстних вікон LLM.

## 2. Ресурсоорієнтована ввічливість (Resource Politeness)

Memory OS розроблений для фонового запуску на машині розробника. Він поважає ресурси хост-системи:
* **Адаптивні пакети (Batches)**: Під час імпорту логів або компактізації пам'яті розмір пачки запитів коригується відповідно до режиму ресурсів (`resource_mode`):
  * `quiet`: Обробка по 2 капсули, обов'язкова пауза 5 секунд між запитами (для економії CPU/RAM та запобігання rate limits).
  * `normal`: Обробка по 5 капсул.
  * `max`: Обробка по 10 капсул паралельно для максимальної швидкості.
* **Локальне збереження даних**: Вся інформація зберігається локально у SQLite БД та JSONL-файлах. Хмарні провайдери не мають доступу до коду, окрім коротких стислих зрізів під час компактізації.

## 3. 12-рівнева шкала квантування складності задач

Складові роботи AI-агента оцінюються за шкалою складності від **L0** до **L13** (основна робоча шкала складається з 12 кроків, L1..L12, а L0 та L13 є граничними випадками).

Квантування задачі виконується за допомогою функції [calculate_score](https://github.com/OlexiMakee/memory_os_public/blob/main/src/memory_os/toolkit/quantizer.py#L40):
1. **Аналіз ключових слів**:
   * Зміна БД / Міграція (`migration`, `schema`, `db`, `database`, `sql`, `table`): **+35 балів** до базової оцінки (10 балів).
   * Архітектурні зміни / Рефакторинг (`refactor`, `solid`, `srp`, `ocp`, `architecture`): **+25 балів**.
   * Робота з безпекою та паролями (`credential`, `encrypt`, `secret`, `password`, `key`): **+25 балів**.
   * Робота з тестами/компіляцією (`test`, `unittest`, `compile`): **+5 балів**.
   * Дрібні правки (`typo`, `todo`, `fix comment`): **-5 балів**.
2. **Вплив факторів ризику (0.0 .. 1.0)**:
   * `risk * 40.0`
   * `volume * 20.0`
   * `uncertainty * 15.0`

Отриманий бал (1-100) конвертується у відповідний рівень складності (Level):

| Діапазон балів | Рівень (Level) | Назва кроку (Step Name) | Модельна політика (Model Policy) | Ідентифікатор воркфлоу (Workflow ID) |
| :--- | :--- | :--- | :--- | :--- |
| **Спеціальний** | L0 | deterministic | `none` | `deterministic.l0` |
| 1 – 10 | L1 | nano | `cheap_free` | `bounded.l1_l4` |
| 11 – 18 | L2 | micro | `cheap_free` | `bounded.l1_l4` |
| 19 – 25 | L3 | tiny | `cheap_free` | `bounded.l1_l4` |
| 26 – 33 | L4 | little | `cheap_free` | `bounded.l1_l4` |
| 34 – 40 | L5 | pretty little | `codex` | `progressive.l5_l11` |
| 41 – 48 | L6 | light mid | `codex` | `progressive.l5_l11` |
| 49 – 56 | L7 | mid | `codex` | `progressive.l5_l11` |
| 57 – 64 | L8 | high mid | `codex` | `progressive.l5_l11` |
| 65 – 75 | L9 | mid high | `strong_cloud` | `progressive.l5_l11` |
| 76 – 82 | L10 | big | `strong_cloud` | `progressive.l5_l11` |
| 83 – 90 | L11 | large | `strong_cloud` | `progressive.l5_l11` |
| 91 – 100 | L12 | giant | `large_reasoning` | `unlimited.l12` (Escalate) |
| **Спеціальний** | L13 | external | `large_reasoning` | `external.l13` (Escalate) |

## 4. Граничні випадки: L0 та L13

### Рівень L0 (Deterministic)
Активується автоматично, якщо фактори ризику дорівнюють нулю, а задача містить виключно ключові слова автоматичного редагування, такі як `lint`, `format` або `syntax`.
* **Стратегія**: Повністю обходить LLM. Використовуються локальні інструменти (наприклад, `black`, `flake8`, `eslint`).

### Рівень L13 (Protocol 13)
Активується автоматично, якщо показник невизначеності (`uncertainty`) перевищує **0.9**.
* **Стратегія**: Повна ескалація. Протокол 13 вимагає ігнорувати поточну базу знань Memory OS, оскільки задача виходить далеко за рамки накопиченого досвіду. Агент повинен аналізувати кодову базу виключно "з нуля" зовні.
